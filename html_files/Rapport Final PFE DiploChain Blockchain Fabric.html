1. Introduction et Contexte du Projet

DiploChain est une plateforme de diplômes numériques sécurisés par blockchain. Elle permet aux institutions académiques d'émettre des diplômes numériques ancrés dans Hyperledger Fabric 2.5, stockés sur IPFS, et vérifiables par les entreprises sans intermédiaire ni tiers de confiance.

Le projet répond à un problème critique : la fraude aux diplômes académiques, qui touche un nombre croissant d'institutions et d'employeurs. DiploChain propose une solution technique robuste basée sur trois piliers :

  • Hyperledger Fabric 2.5 : ancrage immuable on-chain
  • IPFS : stockage décentralisé des PDFs de diplômes
  • Architecture 15 microservices FastAPI : coordonnée par un API Gateway

Architecture principale : 1 base PostgreSQL 14.2 centralisée (diplochain_db) partagée par 15 microservices développés + pdf-generator-service (service existant consommé directement). Le fichier init.sql (pgAdmin 4 ERD) définit 30 tables avec le pattern Base + Extension _ext.

Corrections majeures vs rapports précédents :
  • PostgreSQL 14.2 (pas 16.11)
  • 30 tables (pas 12)
  • PKs serial INTEGER (pas UUID généralisé)
  • Pattern _ext (User + user_ext, institution + institution_blockchain_ext...)
  • 15 microservices développés + pdf-generator-service (existant)
  • channel-id avec tiret (convention Fabric 2.5)

2. Objectifs et Enjeux

Objectifs principaux

  • Éliminer la fraude aux diplômes grâce à une double ancre cryptographique (SHA-256 + CID IPFS)
  • Fournir une vérification instantanée sans intermédiaire via QR code et Hyperledger Fabric
  • Gérer le cycle de vie complet des diplômes (émission, révocation, modification)
  • Assurer la conformité RGPD via des canaux Fabric isolés par institution
  • Offrir un dashboard d'administration en temps réel

Enjeux techniques

  • Scalabilité : architecture microservices permettant une montée en charge indépendante
  • Résilience : Retry Worker pour les diplômes en échec blockchain
  • Sécurité : RBAC, JWT RS256, Azure AD, TLS 1.3
  • Migration future : Database per Service (9 bases déjà créées dans init.sql)

4. Architecture Technique Détaillée

[ Admin / Étudiant / Entreprise ] | v (HTTPS/JWT) +--------------------------------+ | React Dashboard (Frontend) | +---------------+----------------+ | v (REST API + JWT RS256) +---------------+----------------+ +---------------------------+ | API Gateway — FastAPI (8000) | <--> | PostgreSQL 14.2 | +---------------+----------------+ | diplochain_db — 30 tables | | +---------------------------+ +-------+-------+ | | v v [15 Microservices] [Fabric Network] FastAPI Python Hyperledger 2.5.9 | | v v IPFS [Orderer + Peer] (PDF Storage) [CouchDB State]

4 Principes architecturaux fondamentaux

  1. Pipeline IPFS-first : le PDF est stocké sur IPFS avant tout ancrage Fabric
  2. Double ancre cryptographique : hash_sha256 UNIQUE + ipfs_cid enregistrés on-chain
  3. Vérification indépendante de PostgreSQL : le verification-service interroge Fabric comme source de vérité
  4. Résilience via Retry Worker : les diplômes PENDING_BLOCKCHAIN sont re-soumis avec backoff exponentiel

Table diplome_blockchain_ext (Clé Blockchain)

Cette table est le cœur du système, liant les données métier à la blockchain.

  • id_diplome (PK FK CASCADE) : Lien vers etudiant_diplome
  • hash_sha256 (VARCHAR(64) UNIQUE) : Empreinte numérique unique du PDF
  • tx_id_fabric (VARCHAR) : Transaction ID Fabric (NULL si PENDING)
  • ipfs_cid (VARCHAR) : Content ID du fichier sur IPFS
  • statut (ENUM) : État du cycle de vie
  • blockchain_retry_count : Compteur pour le worker de résilience

7. Pipeline d'Émission de Diplôme

Ordre obligatoire du pipeline : pdf-generator-service ⚡ → storage-service → diploma-service (INSERT PENDING) → blockchain-service (Fabric) → diploma-service (UPDATE ORIGINAL) → notification → analytics

# Action détaillée Service Table écrite Statut
1 Admin soumet formulaire émission (mode: UPLOAD ou GENERATE) Frontend → api-gateway → diploma-service — Validation champs
2 Récupération profil étudiant (email, specialite, CIN...) diploma → student-service GET /students/{id} etudiant (lecture) Données étudiant
3 Mode GENERATE : récupération template_departement actif diploma → institution-service GET /institutions/{id}/template template_departement (lect.) jrxml_path, jasper_path
4 Mode GENERATE : génération PDF Jasper → SHA-256 calculé diploma → pdf-generator-service ⚡ EXISTANT POST /generate — (binaire en mémoire) PDF binaire + hash
5 Upload PDF vers IPFS → CID reçu diploma → storage-service POST /storage/upload diplome_blockchain_ext.ipfs_cid CID IPFS retourné
6 INSERT etudiant_diplome (métadonnées officielles) diploma-service etudiant_diplome id_diplome créé
7 INSERT diplome_blockchain_ext (statut=PENDING_BLOCKCHAIN) diploma-service diplome_blockchain_ext PENDING_BLOCKCHAIN

9 UPDATE diplome_blockchain_ext SET tx_id_fabric + statut=ORIGINAL diploma-service diplome_blockchain_ext ORIGINAL

8. Retry Worker Service

Le retry-worker-service est un processus autonome qui surveille les diplômes bloqués en statut PENDING_BLOCKCHAIN (suite à une panne réseau Fabric par exemple) et les soumet à nouveau automatiquement.

Étape Détail
Détection PENDING SELECT * FROM diplome_blockchain_ext WHERE statut='PENDING_BLOCKCHAIN' ORDER BY created_at ASC LIMIT 50
Filtre retry blockchain_retry_count < max_retries (défaut 5) AND blockchain_last_retry < NOW() - INTERVAL '15 min'
Re-soumission Fabric POST blockchain-service/blockchain/register
Succès UPDATE SET tx_id_fabric, statut='ORIGINAL' + INSERT historique_operations
Échec UPDATE SET blockchain_retry_count = blockchain_retry_count + 1, blockchain_last_retry = NOW()
Alerte max retry Si retry_count >= max_retries → alerte admin-dashboard-service
Scheduler CRON toutes les 15 minutes via APScheduler Python

9. Flux de Vérification

La source de vérité pour la vérification est le ledger Hyperledger Fabric. PostgreSQL n'est PAS utilisé pour valider l'authenticité.

# Action Service Source de données
1 Scan QR Code → identifiant_opaque reçu Frontend / App mobile URL publique → api-gateway
2 Résolution identifiant_opaque → diplome_id verification-service → qr_code_records qr_code_records.identifiant_opaque UNIQUE
3 Validation JWT Azure AD (si vérification entreprise) api-gateway → entreprise-service entreprise_auth_sessions (is_valid + expires_at)
4 QueryDiploma(diplome_id) → hash_sha256 + ipfs_cid on-chain verification → blockchain-service Hyperledger Fabric ledger (JAMAIS PostgreSQL)
5 Téléchargement PDF via CID IPFS verification → storage-service GET /storage/download/{cid} IPFS — CID récupéré depuis Fabric
6 Recalcul SHA-256 du PDF téléchargé verification-service (hashlib.sha256()) En mémoire — aucune lecture BDD
7 Comparaison hash recalculé vs hash on-chain Fabric verification-service VALIDE si hash == hash_fabric. INVALIDE sinon.
8 INSERT historique_operations (type_operation=VERIFICATION) verification-service historique_operations — audit uniquement
9 Retour JSON {valid, details, etudiant, institution} api-gateway → Frontend / Entreprise Réponse verification-service

12. Hyperledger Fabric — Guide Complet

12.1 Pourquoi Hyperledger Fabric ?

Critère Fabric (choisi) Ethereum public
Confidentialité Canaux isolés — données d'une institution invisibles aux autres Tout est public par défaut
Performance ~1000 TPS, finalité en secondes ~15 TPS sur mainnet, finalité probabiliste
Identité Certificats X.509, MSP, rôles précis Clés cryptographiques anonymes
Coût Pas de gas fees — réseau privé Gas fees variables et coûteux
Contrôle Gouvernance privée — règles personnalisables Règles immuables du protocole public
RGPD Données on-chain contrôlées, révocables Immuabilité totale problématique pour RGPD

12.6 Workflow d'installation Fabric (6 étapes)

Étape 1: Installation binaires Fabric 2.5.9

curl -sSL https://github.com/hyperledger/fabric/releases/download/v2.5.9/hyperledger-fabric-linux-amd64-2.5.9.tar.gz | tar -xz -C .
docker pull hyperledger/fabric-peer:2.5
docker pull hyperledger/fabric-orderer:2.5
echo 'export PATH=$HOME/diplochain/fabric-network/bin:$PATH' >> ~/.bashrc

Étape 2: Génération des certificats (cryptogen)

export FABRIC_CFG_PATH=$PWD
cryptogen generate --config=./crypto-config.yaml --output=./crypto-config

Étape 3: Génération du genesis block

configtxgen -profile DiploChainGenesis -channelID diplochain-system-channel -outputBlock ./channel-artifacts/genesis.block

Étape 4: Démarrage conteneurs Docker

docker-compose -f docker-compose.fabric.yml up -d
docker ps --filter 'name=diplochain' --format 'table {{.Names}}\t{{.Status}}'

Étape 5: Création canal institution

./scripts/create-channel.sh 1   # → crée channel-1 pour institution_id=1
./scripts/create-channel.sh 2   # → crée channel-2 pour institution_id=2

Étape 6: Déploiement chaincode (4 phases)

# Phase 1: Package
peer lifecycle chaincode package ./channel-artifacts/diplochain.tar.gz --path ./chaincode/diplochain --lang golang --label diplochain_1.0
# Phase 2: Install
peer lifecycle chaincode install /tmp/diplochain.tar.gz
# Phase 3: Approve
peer lifecycle chaincode approveformyorg -o orderer.diplochain.local:7050 --channelID channel-1 --name diplochain --version 1.0 --sequence 1
# Phase 4: Commit
peer lifecycle chaincode commit -o orderer.diplochain.local:7050 --channelID channel-1 --name diplochain --version 1.0 --sequence 1

15. Limitations et Perspectives

Limitations

  1. Base partagée diplochain_db : risque de couplage — Migration Database per Service planifiée (9 bases créées dans init.sql)
  2. PKs serial (INTEGER) : si migration UUID requis, impact sur toutes les FK et APIs REST
  3. Incohérence types PK : template_departement.departement_id est UUID mais FKs institutions sont en INTEGER
  4. Cohérence éventuelle : transactions distribuées peuvent partiellement échouer — Saga pattern requis
  5. notification-service sans file : si le service tombe, emails perdus — Redis/RabbitMQ recommandé

Perspectives

  1. Migration Database per Service : isoler chaque groupe dans sa propre base
  2. Event-driven architecture : Kafka pour découpler et garantir livraison notifications
  3. Service Mesh (Istio) : observabilité, circuit breakers, mutual TLS
  4. Migration UUID généralisée : aligner tous les PKs en UUID
  5. API publique de vérification : endpoint public sans authentification pour vérification rapide QR
  6. Mobile app : application mobile pour étudiants et entreprises

16. Conclusion

DiploChain implémente une architecture de 15 microservices développés + pdf-generator-service (service existant) s'appuyant sur PostgreSQL 14.2 centralisée (diplochain_db) contenant 30 tables organisées selon le pattern Base + Extension _ext.

4 principes architecturaux fondamentaux :
  1. Pipeline IPFS-first — le PDF est stocké sur IPFS avant l'ancrage Fabric
  2. Double ancre cryptographique — hash_sha256 UNIQUE + ipfs_cid dans diplome_blockchain_ext
  3. Vérification indépendante de PostgreSQL — le Verification Service interroge Fabric (source de vérité on-chain)
  4. Résilience via Retry Worker — les diplômes en échec Fabric (PENDING_BLOCKCHAIN) sont automatiquement re-soumis
